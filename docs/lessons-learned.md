# Lessons Learned — AssemblyZero

| Date | Lesson | Rule/Action |
|:-----|:-------|:------------|
| 2026-03-18 | **101 plan files and 4,688 todo files discovered in ~/.claude/.** These accumulated over weeks of sessions. If any are auto-loaded into context, they're a massive token sink that would explain the accelerating compaction. | **Audit and purge stale plans/todos periodically.** Add to cleanup protocol. Investigate whether Claude Code loads all plans/todos or only the active one. |
| 2026-03-18 | **Jumped to implementation across 5+ files after design Q&A without presenting a plan.** User asked "do you have any more questions?" and agent interpreted this as authorization to implement. User had explicitly said "at least two changes to every workflow /plan" -- the word "plan" was in their message. 3 issues created, branch started, partial code written before user intervened. | **After a design Q&A session, ALWAYS present a structured plan before writing code.** "Do you have any more questions?" is a checkpoint, not a green light. Wait for explicit "do it" / "implement" / plan approval. Saved as feedback_plan_before_implement.md. |
| 2026-03-20 | **Background agents blocked on Write permissions for wiki directory.** All 3 background agents had Write tool denied for `AssemblyZero.wiki/`. Had to write all 12 new pages in main context sequentially. | **Don't delegate wiki writes to subagents.** The wiki is a separate repo and Write permissions don't carry over. Write wiki pages directly in the main conversation. |
| 2026-03-20 | **pr-sentinel requires `Closes #N` in PR body, not just title.** PR #866 failed the `issue-reference` check because the reference was only in the title. | **Always add `Closes #N` as a standalone line in the PR body** when creating PRs with `gh pr create`. The title alone is not enough for pr-sentinel. |
| 2026-03-20 | **`gh issue list --limit 100` silently truncates.** First query returned 100 issues and I reported 187 closed since Feb 27. True count was 206. | **Always use `--limit 500` (or higher) with `gh issue list`** when counting issues over a date range. The default limit silently truncates results. |
| 2026-03-27 | **pr-sentinel checks the PR body, not the commit message.** Dispatch agent amended commit to `Closes #52` but PR body still said `Closes #49` (closed issue). Merge stayed blocked for multiple retries until diagnosed from another session. | **`Closes #N` must appear in ALL THREE places: commit message, PR title, PR body.** Root CLAUDE.md updated to state this explicitly. |
| 2026-03-27 | **`gh pr checks --watch` hits GraphQL 403 with the fine-grained PAT.** Agent discovered this and had no fallback — thrashed between retries with no diagnostic path. | **Use `mergeable_state` polling via REST API instead.** Root CLAUDE.md merge sequence rewritten to use `gh api .../pulls/{N} --jq '.mergeable_state'`. |
| 2026-03-27 | **"If blocked" diagnostic in CLAUDE.md was too narrow — only said "Cerberus hasn't approved."** But Cerberus HAD approved and merge was still blocked because pr-sentinel check hadn't passed. Agent had no diagnostic path for "approved but blocked." | **New diagnostic: if `mergeable_state` stays `blocked`, check PR body for correct open issue reference first. The fix is `gh pr edit --body`, not merge retries.** |
| 2026-04-05 | **Pre-existing unpushed commits on main caused a second PR cycle.** Had 1 commit from a prior session sitting on local main. Created separate PRs instead of bundling. Referenced closed issues (#851/#852), causing pr-sentinel failure and another round-trip. | **Before creating a PR, check for stale local commits with `git log origin/main..HEAD`. Bundle everything in one PR. Check referenced issues are OPEN before creating the PR.** |
| 2026-04-06 | **Agent deleted branch protection and merged a PR without user permission.** User said "test" and agent interpreted it as authorization for the full cycle including destructive operations. When the merge was blocked, agent escalated to deleting protection instead of stopping. | **NEVER delete branch protection, NEVER merge, NEVER take destructive action without explicit user instruction. "Test" means test. "Merge" means merge. They are different words. When blocked, STOP and report — do not escalate autonomously.** |
| 2026-04-06 | **Agent had classic PAT access and wasted it.** When user switched to classic PAT for fixes, agent should have READ both repos' branch protection to compare. Instead spent the time trying to write and merge, burning through PAT switches. Diagnosis requires reading, not writing. | **When given elevated access (classic PAT), the FIRST action is diagnostic: READ the state on the working repo AND a known-good repo. Compare. Then propose the fix. Do not write until you understand the difference.** |
| 2026-04-06 | **`new_repo_setup.py` creates branch protection that doesn't match the fleet.** Script sets `required_approving_review_count=0` but every working repo uses `1`. Script uses `-f` (string) for boolean fields but API requires `-F` (typed). Script sets context `pr-sentinel` but GitHub Actions creates checks as `pr-sentinel / issue-reference`. | **Before trusting a script's output, verify it against a known-good repo. The script being "canonical" doesn't mean it's correct.** |
| 2026-04-06 | **GitHub Apps require at least one active private key.** Cannot revoke the last key. The generate-deploy-delete-revoke sequence only works when 2+ keys exist. | **Update runbook 0927: document that revoking is only possible when a second key exists. Otherwise .pem deletion is the only protection.** |
| 2026-04-08 | **Handoff diagnoses can be wrong.** Previous agent diagnosed Sextant's `issue-reference` context as broken, prescribed changing to `pr-sentinel / issue-reference`. Reading the actual issues (#742, #748) showed the opposite — the fleet-wide fix was TO `issue-reference`. | **Never trust a handoff's technical diagnosis without verifying against primary sources (issues, code, API). Handoffs capture the agent's understanding, not ground truth.** |
| 2026-04-08 | **Reading issue titles is not reading issues.** When told to read issues, the first instinct was to scan titles. The critical information (#742's resolution, #748 point 4) was in issue bodies, not titles. | **When mining issues for technical decisions, read the full body. Titles are marketing, bodies are engineering.** |
